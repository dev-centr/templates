= VSCode Workspace Configuration

The `templates` repository provides workspace configurations and boilerplate files for modern development projects. It contains both template files that should be copied to new repositories and active configuration files that can be included in workspace files.

== Repository Structure

The `templates` repository is organized as follows:

* Root level: Files specific to the templates repository itself
* `workspaces/`: Contains VS Code workspace templates for different workflows and configurations

== Workspace Templates

The `workspaces/` folder contains template folders that serve dual purposes:

TIP: Each template folder now includes a `README.adoc` with usage notes, extension lists, and workspace instructions. Open the README directly inside `workspaces/+{name}+` when you need a refresher on what a template provides.

=== Template Files (Copy to New Repos)

Each template folder contains files that should be copied to new repositories:

* `.gitattributes` - Git line ending configuration
* `.gitignore` - Git ignore patterns
* `.hintrc` - Webhint configuration
* `.github/FUNDING.yml` - GitHub funding configuration

=== Active Configuration Files (Include in Workspaces)

Each template folder also contains active configuration files that can be copied into your project workspace:

* `+{template}+.code-workspace` - Combined settings and extension recommendations

=== Available Templates

* `_common/` - Common extensions and template files for all projects (docs, git, media tools)
* `astro/` - Astro specific extensions and optimized settings
* `javascript/` - JavaScript/TypeScript workflow extensions (Biome, dprint)
* `next.js/` - Next.js specific extensions (Next.js assistant, icons, snippets)
* `python/` - Python workflow extensions (Ruff formatter)
* `rust/` - Rust workflow extensions
* `solidstart/` - SolidStart workflow extensions

== Using Templates in Workspaces

To use a template in your project workspace, we recommend copying the relevant blocks from **both** the `_common` template and your specific tech stack template (e.g., `javascript/` or `next.js/`) directly into your project's `.code-workspace` file.

Each template folder contains its own `.code-workspace` file pre-configured with industry-standard settings and extension recommendations.

1. Open the `common.code-workspace` file in `_common/` and copy the `"settings"` and `"extensions"` blocks.
2. Open the `.code-workspace` file within your desired tech stack folder (e.g., `astro.code-workspace`).
3. Merge its `"settings"` and `"extensions"` blocks into your project's workspace file, being careful to combine rather than overwrite the arrays and objects.

Copying these blocks directly into your project ensures a self-contained and fully configured development environment that matches the standards defined in the Dev Center, without the complexity of managing relative folder paths in your workspace definition.

== Global Workspace

The `workspaces/global.code-workspace` file provides an example workspace definition that includes the template repository itself, useful as a starting point for creating your own multi-root workspaces.

== Settings and Extension Recommendations

**Important:** The `.vscode/settings.json` and `.vscode/extensions.json` files do NOT automatically apply to other repositories when included in a multi-root workspace. 

To share settings and extension recommendations across repositories, you must embed them directly in your project's `.code-workspace` file. We provide pre-configured `.code-workspace` files in each template folder that you can copy from.

[WARNING]
====
**Settings Precedence & Priority:** VS Code settings follow a strict hierarchy of visibility: **User Settings > Workspace Settings (.code-workspace) > Folder Settings (.vscode/settings.json)**.

However, **Language-specific settings** (e.g., `[json]`) always have higher priority than **Global settings** (e.g., `editor.defaultFormatter`), even if they are defined at a lower-priority scope. 

For example, if a user has a "User" level language default set for JSON (like Prettier), it will override a "Workspace" level global default (like Biome). To ensure project standards are enforced, we use explicit language-level blocks in our workspace files to provide the necessary specificity to win against personal defaults.
====

== Creating New Templates

When creating a new template folder:

1. Create a folder named after the technology or workflow (e.g., `+{workflow}+/`) in `workspaces/`
2. Add a `README.adoc` explaining the template's purpose and contents
3. Add a `+{workflow}+.code-workspace` file with the relevant `"settings"` and `"extensions"` blocks
4. Optionally add template files (`.gitattributes`, `.gitignore`, etc.) that should be copied to new repos

[IMPORTANT]
====
**Extension Sourcing:** We only list extensions that are available on the community-run link:https://open-vsx.org[OpenVSX marketplace]. That guarantees they can be installed in Cursor and other third-party VS Code forks without extra setup.
====

Template folders are named after the technology or workflow they support, making it clear what the template is for when seen in the workspace.
