= Common Template

The `_common` workspace template provides baseline tooling that every repository can rely on. Add it to any VS Code workspace when you want the shared documentation, git, and formatting helpers without copying files into your project.

== When to include it

* You are working in a multi-repo workspace and want consistent Git tooling (GitHub Pull Requests, GitLens, Git Graph).
* You need the shared documentation stack (dprint, Biome, Mermaid, AsciiDoc, SVG editor).
* You want YAML schema hints for `.github/FUNDING.yml` and other shared automation files.
* You use Antora for documentation and want standard `antora-playbook.yml` / `antora-playbook.testing.yml` ignore rules.

== Included configuration

* `.vscode/extensions.json` – curated list of cross-cutting extensions. They all ship on the community-run OpenVSX marketplace, so they install cleanly in Cursor and other VS Code forks.
* `.vscode/settings.json` – shared YAML schema mappings, formatter defaults, and other base tweaks that are safe to merge into any repository.

[IMPORTANT]
====
**Settings Precedence:** When you open a single folder, VS Code applies **User** settings first, then **Folder** (`.vscode/settings.json`). **Language-specific settings** (e.g., `[json]`) override global keys at the same scope.

To ensure project standards "win" against personal user defaults (like a global Prettier preference), we use explicit language-level overrides in `.vscode/settings.json`. If you find your personal settings are interfering, ensure your folder-level language overrides are correctly defined.
====

== How to use it

To use this template, copy `.vscode/settings.json` and `.vscode/extensions.json` from this folder into your project (merge objects and recommendation arrays with care).

[IMPORTANT]
====
You should also merge in the `.vscode` files from your chosen tech stack template (e.g., `javascript/`) so formatter rules, excludes, and extension lists stay complete.
====

Copying or merging these files into your project ensures that your development environment is fully configured with the shared standards of the Dev Center, without needing to reference external folders.

If you need to copy a specific configuration file (for example `.github/FUNDING.yml`), open the template folder directly and copy the file into your repo.
